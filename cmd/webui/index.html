Distributed Rate Limiter

Submit a request and review the rate-limit decision.

This interface submits requests through the HTTP gateway, forwards them through Envoy and gRPC, and returns a Redis-backed rate-limit decision. Every decision is also published to a Redis Stream and delivered in real time to the Live Event Stream. The system ships structured logs to Loki, distributed traces to Tempo, and fires alerts through AlertManager — all visible in the Debug Dashboard and Grafana.

1. Browser Define the request and submit it from this page.
2. Web UI The HTTP gateway converts the request into a gRPC call.
3. Envoy Forwards gRPC traffic across all rate-limiter replicas.
4. Rate Limiter Parses the rule, runs the algorithm, publishes an event.
5. Redis Stores shared counters and the rl:events stream.
6. Streamer Reads the stream and fans out to WebSocket clients.
7. Observability Prometheus · Loki · Tempo · AlertManager · Grafana.
What These Options Mean Field definitions for the request form.
namespace Keeps unrelated traffic from sharing the same rate limit.
key Identifies the client or entity subject to the rate limit.
rule Defines the limit as either a refill rate or a requests-per-window threshold.
algorithm Selects token bucket or sliding window. AUTO derives the algorithm from the rule format.
cost Applies a multi-unit charge when a single request should consume additional budget.
Algorithm Comparison When to choose token bucket versus sliding window.
Token bucket Smoother and burst-friendly. This is a strong choice for general API throughput control.
Sliding window Stricter and more exact within a time window. This is a strong choice for login, OTP, or abuse protection.
